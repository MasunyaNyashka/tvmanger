# Нагрузочные тесты

## Назначение

Каталог `load-tests` содержит сценарии нагрузочного тестирования API на базе `k6`.

Сценарии рассчитаны на тестовое окружение, поднимаемое через `docker-compose.test.yml`, и используются для первичной оценки:

- устойчивости сервисов под параллельной нагрузкой;
- времени ответа (`p95`, `p99`);
- доли неуспешных запросов;
- количества ошибок в бизнес-операциях.

## Текущий сценарий

Файл `load-tests/api-load.js` выполняет следующий поток:

1. регистрация клиента;
2. логин клиента;
3. получение публичного списка тарифов;
4. создание заявки на подключение;
5. создание сервисной заявки.

Сценарий представляет собой базовый `smoke/load` профиль и не является полноценным stress- или soak-тестом.

## Предварительные требования

Перед запуском нагрузочных тестов необходимо:

1. Поднять тестовый стенд:

```bash
docker compose -f docker-compose.test.yml up --build
```

2. Установить `k6`, если он ещё не установлен.

Пример для Windows:

```powershell
winget install k6 --source winget
```

## Параметры окружения

По умолчанию используются следующие адреса сервисов:

```text
AUTH_BASE_URL=http://localhost:18081
TARIFF_BASE_URL=http://localhost:18082
ORDER_BASE_URL=http://localhost:18083
SERVICE_REQUEST_BASE_URL=http://localhost:18084
```

При необходимости адреса могут быть переопределены через переменные окружения `k6`.

## Запуск

Базовый запуск:

```bash
k6 run load-tests/api-load.js
```

Запуск с явным указанием адресов:

```bash
k6 run ^
  -e AUTH_BASE_URL=http://localhost:18081 ^
  -e TARIFF_BASE_URL=http://localhost:18082 ^
  -e ORDER_BASE_URL=http://localhost:18083 ^
  -e SERVICE_REQUEST_BASE_URL=http://localhost:18084 ^
  load-tests/api-load.js
```

## Профиль нагрузки

Текущий профиль использует `ramping-vus`:

- старт с `1` виртуального пользователя;
- увеличение до `5` пользователей за `30s`;
- увеличение до `10` пользователей за `1m`;
- снижение до `0` пользователей за `30s`.

Пороговые значения:

- `http_req_failed < 10%`
- `http_req_duration p(95) < 1500ms`
- `http_req_duration p(99) < 3000ms`
- `successful_flows > 80%`

## Основные метрики

При анализе результатов следует ориентироваться на:

- `http_req_failed`
- `http_req_duration`
- `p(95)`
- `p(99)`
- `business_errors`
- `successful_flows`
